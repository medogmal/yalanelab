# تقرير فحص المنصة (Platform Audit Report) - Yalla Nelab

**التاريخ:** 06 مارس 2026
**الحالة العامة:** 🟡 نموذج أولي متقدم (Advanced Prototype) - غير جاهز للإنتاج الفعلي (Not Production Ready)

---

## 🚨 ملخص النواقص الحرجة (Critical Gaps)
المنصة حالياً تعمل كنموذج عرض (Demo) يعتمد على الذاكرة المؤقتة (RAM) والتخزين المحلي (LocalStorage). لإطلاقها كمنتج فعلي، يجب معالجة النقاط التالية بشكل جذري:

1.  **قاعدة البيانات (Database):** غير موجودة. جميع البيانات (المستخدمين، الأرصدة، المباريات) تُفقد عند إعادة تشغيل الخادم.
2.  **نظام اللعب الجماعي (Online Multiplayer):** يعتمد على ذاكرة الخادم المؤقتة (In-Memory) ولا يوجد خادم "Socket.IO" حقيقي، مما يعني أن اللعب سيفشل عند نشر الموقع على Vercel أو أي بيئة Serverless.
3.  **الأمان والتوثيق (Security & Auth):** لا يوجد نظام تحقق حقيقي (JWT/Session). يمكن لأي شخص تزوير هويته أو رصيده بسهولة.

## 1. البنية التحتية والواجهة الخلفية (Backend & Infrastructure)

| المكون | الحالة الحالية | المطلوب للإنتاج | الأولوية |
| :--- | :--- | :--- | :--- |
| **قاعدة البيانات** | `store.ts` (مصفوفات في الذاكرة) | ربط قاعدة بيانات حقيقية (PostgreSQL / MongoDB / Redis). | 🔴 حرجة |
| **التوثيق (Auth)** | `localStorage` (بدون تحقق) | نظام توثيق آمن (NextAuth.js / Clerk / Firebase Auth). | 🔴 حرجة |
| **الخادم (Server)** | Next.js API Routes (Serverless) | خادم مخصص للألعاب (Custom Node.js Server) لدعم الـ WebSockets المستمرة. | 🔴 حرجة |
| **المدفوعات** | وهمية (Mock) | ربط بوابات دفع حقيقية (Stripe / PayTabs / Tap). | 🟠 متوسطة |

### 🛠️ التوصية التقنية:
*   استخدام **Prisma + PostgreSQL** لحفظ بيانات المستخدمين والمباريات.
*   استخدام **Redis** لحفظ حالة المباريات السريعة (Game State) بدلاً من ذاكرة الخادم.
*   فصل خادم الألعاب (Socket Server) عن خادم الموقع (Next.js) لضمان الأداء.

## 2. حالة الألعاب (Games Status)

### 🎲 لودو (Ludo)
*   **المنطق:** مكتمل بنسبة 80% (الحركة، النرد، القتل).
*   **الأونلاين:** يستخدم `global.__LUDO_MEM__` وهو غير صالح للإنتاج. يحتاج لنظام غرف (Rooms) عبر Socket.IO + Redis.
*   **الذكاء الاصطناعي:** موجود ولكنه بسيط (عشوائي أحياناً).

### ♟️ شطرنج (Chess)
*   **المنطق:** ممتاز (يعتمد على `chess.js` و `stockfish`).
*   **الأونلاين:** نفس مشكلة اللودو (ذاكرة مؤقتة). يحتاج لمزامنة الحركات عبر السوكيت.
*   **المظهر:** جيد جداً مع دعم الاسكنات.

### ♠️ بلوت (Baloot)
*   **المنطق:** القوانين الأساسية موجودة، لكن نظام المزايدة (Bidding) وحساب النقاط (Projects) يحتاج لاختبارات مكثفة للتأكد من تغطية كل الحالات (Sra, Baloot, 400, etc).
*   **الأونلاين:** غير جاهز فعلياً (تعتمد على Polling أو محاكاة).
*   **الذكاء الاصطناعي:** يحتاج لتحسين كبير ليكون منافساً حقيقياً (حالياً يلعب بشكل قانوني ولكن ليس ذكياً).

### 🁙 دومينو (Domino)
*   **المنطق:** جيد (حساب النقاط، القفلة).
*   **الأونلاين:** غير جاهز للإنتاج.

## 4. خطة العمل المقترحة (Roadmap to Production)

### المرحلة 1: التأسيس (أسبوعين)
1.  تنصيب قاعدة بيانات **PostgreSQL**.
2.  تفعيل **NextAuth** وتسجيل الدخول الحقيقي (Google/Email).
3.  نقل بيانات المستخدمين من `localStorage` إلى قاعدة البيانات.

### المرحلة 2: الخادم الحي (3 أسابيع)
1.  إعداد خادم **Socket.IO** منفصل (أو مدمج بـ Custom Server).
2.  إعادة كتابة منطق الألعاب (Game Server Logic) ليعتمد على الـ Sockets بدلاً من الذاكرة المؤقتة.
3.  استخدام **Redis** لتخزين حالة الطاولات النشطة.

### المرحلة 3: التحسين والإطلاق (أسبوعين)
1.  ربط المتجر ونظام العملات بقاعدة البيانات.
2.  تحسين الذكاء الاصطناعي (Bot Difficulty Levels).
3.  اختبار الحمل (Load Testing) للتأكد من تحمل عدد كبير من اللاعبين.
